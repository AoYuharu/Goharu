# 提示词工程亮点分析

## 🎯 提示词工程的核心价值

在 AI 应用开发中，提示词工程往往决定了系统的实际效果。这个项目在提示词设计上有多个创新点。

---

## 🌟 核心创新点

### 1. **结构化提示词系统**（PromptDocument + PromptSection）

**创新点**：
- 将提示词拆分为独立的 Section，每个 Section 有明确的职责
- 支持灵活组装，不同场景使用不同的 Section 组合
- 每个 Section 可以独立缓存

**技术实现**：
```python
class PromptSection:
    kind: str  # system, user, assistant, tool
    title: str
    content: str
    cache_control: dict  # 支持 Prompt Caching
    metadata: dict

class PromptDocument:
    system_sections: List[PromptSection]
    conversation_sections: List[PromptSection]
```

**面试话术**：
> "我设计了一个结构化的提示词系统。传统做法是把所有提示词拼接成一个大字符串，难以维护和复用。我将提示词拆分为独立的 Section，每个 Section 有明确的职责和元数据。这样可以灵活组装，也方便实现 Prompt Caching。"

**优势**：
- ✅ 可维护性：每个 Section 独立管理
- ✅ 可复用性：不同场景复用相同的 Section
- ✅ 可测试性：可以单独测试每个 Section
- ✅ 可缓存性：每个 Section 可以独立缓存

---

### 2. **分层提示词架构**

**创新点**：
- 将提示词分为 5 层，每层有不同的更新频率和缓存策略
- 从最稳定到最动态：SOUL → Tools → User Profile → Memory → History

**架构设计**：
```
┌─────────────────────────────────────────┐
│ Layer 1: SOUL.md (角色定义)             │  ← 最稳定，始终缓存
│ - 你是谁，你的职责是什么                 │
│ - 核心原则和行为边界                     │
├─────────────────────────────────────────┤
│ Layer 2: Tool Directory (工具目录)      │  ← 最稳定，始终缓存
│ - 可用工具列表和 schema                  │
│ - 工具使用规则和限制                     │
├─────────────────────────────────────────┤
│ Layer 3: User Profile (用户画像)        │  ← 中频更新，默认缓存
│ - 用户偏好和习惯                         │
│ - 历史交互总结                           │
├─────────────────────────────────────────┤
│ Layer 4: Memory (长期记忆)              │  ← 中频更新，默认缓存
│ - 重要事实和知识                         │
│ - 历史任务总结                           │
├─────────────────────────────────────────┤
│ Layer 5: History (对话历史)             │  ← 高频更新，按轮次缓存
│ - 历史轮次：缓存                         │
│ - 最新一轮：不缓存                       │
└─────────────────────────────────────────┘
```

**面试话术**：
> "我设计了一个分层提示词架构。最底层是 SOUL（角色定义），几乎不变，始终缓存。往上是工具目录、用户画像、长期记忆，更新频率逐渐增加。最顶层是对话历史，最新一轮不缓存。这种分层设计既保证了提示词的完整性，又最大化了缓存效率。"

**优势**：
- ✅ 清晰的职责分离
- ✅ 最大化缓存效率（85-90% 成本节省）
- ✅ 灵活的更新策略

---

### 3. **角色定义与行为约束**（SOUL.md）

**创新点**：
- 明确的角色定义：你是谁，你的职责是什么
- 强制性的行为约束：禁止编造、工具优先、执行不描述
- 详细的工具使用指南：什么时候用什么工具

**核心内容**：
```markdown
## 核心原则

**禁止编造结果**：绝对不能编造、假设或猜测任何执行结果。

**工具优先**：当用户要求执行操作时，必须：
1. 先调用工具执行操作
2. 等待工具返回真实结果
3. 基于工具返回的结果回答用户
4. 绝不能在没有调用工具的情况下声称"已完成"

**执行，不要描述**：
- ❌ 错误：说"我将创建一个文件"
- ✅ 正确：直接调用工具执行操作
```

**面试话术**：
> "我在 SOUL.md 中定义了 Agent 的核心行为原则。最重要的是'禁止编造结果'和'工具优先'。很多 Agent 系统的问题是 LLM 会编造执行结果，说'我已经创建了文件'，但实际上没有调用工具。我通过强制性的约束和详细的示例来解决这个问题。"

**难点**：
- 如何让 LLM 真正遵守约束（需要反复强调和示例）
- 如何平衡约束的严格性和灵活性
- 如何处理 LLM 的"幻觉"问题

---

### 4. **工具调用格式的强制约束**

**创新点**：
- 明确规定唯一正确的工具调用格式
- 列举所有错误格式，并说明为什么错误
- 使用多层防护机制确保格式正确

**提示词设计**：
```markdown
## 工具调用格式

**严格要求**：当你需要使用工具时，必须【只输出纯 JSON】。

**唯一正确格式**：
{"tool": "工具名", "arguments": {...}}

**错误格式示例**：
- ❌ 错误：<invoke name="run_cmd">...</invoke>
- ❌ 错误：tool => "run_cmd"
- ❌ 错误：```json\n{"tool": "run_cmd", ...}\n```
- ❌ 错误：我将调用工具：{"tool": "run_cmd", ...}
```

**多层防护**：
```python
# 防线 1: 提示词明确规定格式
# 防线 2: ToolCall.try_from_text() 解析
# 防线 3: ToolCallGuard 修复常见错误
# 防线 4: Schema 验证
# 防线 5: 重试机制（最多 3 次）
```

**面试话术**：
> "工具调用格式是一个常见的痛点。LLM 可能输出各种格式：XML、Markdown 代码块、带前缀的 JSON 等。我设计了多层防护：提示词明确规定格式，解析器容错，Guard 自动修复，Schema 验证，重试机制。这样可以处理 95% 以上的格式问题。"

**难点**：
- LLM 的输出格式不稳定
- 不同模型的行为差异
- 如何在严格约束和容错之间平衡

---

### 5. **上下文感知的提示词注入**

**创新点**：
- 根据当前状态动态调整提示词
- Reflection 反馈会作为 user 消息注入
- 工具执行结果会作为 tool 消息注入

**动态注入示例**：
```python
# 场景 1: Reflection 要求继续调用工具
memory_manager.append({
    "role": "user",
    "content": "[Reflection] 信息不足，请调用 Read 工具读取 config.yaml"
})

# 场景 2: Reflection 要求给出最终回答
memory_manager.append({
    "role": "user",
    "content": "[Reflection] 信息已充分，现在给出最终回答，禁止继续调用工具。"
})

# 场景 3: 工具执行失败
memory_manager.append({
    "role": "user",
    "content": "工具执行失败: 文件不存在。请检查路径并重新调用。"
})
```

**面试话术**：
> "我实现了上下文感知的提示词注入。根据当前状态动态调整提示词，而不是静态的系统提示词。比如 Reflection 判断信息不足时，会注入'请调用 Read 工具'；判断信息充分时，会注入'现在给出最终回答'。这种动态注入比静态提示词更有效。"

**优势**：
- ✅ 更精确的控制
- ✅ 更好的上下文理解
- ✅ 更灵活的行为调整

---

### 6. **Reflection 的审核提示词设计**

**创新点**：
- 明确的审核标准（完整性、准确性、清晰性、相关性）
- 强制输出特定短语（"答案可以接受"或"答案需要改进"）
- 提供工具调用结果和文件内容，但不包含 Actor 思考

**提示词结构**：
```markdown
## 审核标准

1. **完整性**：答案是否完整回答了用户的所有问题点
2. **准确性**：答案是否基于实际的工具调用结果，而不是编造
3. **清晰性**：答案是否清晰易懂，逻辑连贯
4. **相关性**：答案是否紧扣问题，没有偏题

## 审核输出

你必须在回复中明确包含以下短语之一：
- **"答案可以接受"** - 如果答案质量良好
- **"答案需要改进"** - 如果答案存在明显问题

如果答案需要改进，请具体说明问题所在和改进建议。
```

**面试话术**：
> "Reflection 的审核提示词设计很关键。我定义了明确的审核标准，并强制输出特定短语。这样可以通过简单的字符串匹配来判断审核结果，避免解析复杂的自然语言。同时，Reflection 只看工具调用结果，不看 Actor 的思考过程，确保独立判断。"

**难点**：
- 如何设计审核标准（太严格会导致频繁拒绝，太宽松失去意义）
- 如何确保 LLM 输出特定短语（需要反复强调）
- 如何提供足够的上下文（工具调用结果 + 文件内容）

---

### 7. **测试策略的提示词指导**

**创新点**：
- 根据代码类型选择不同的测试策略
- 明确区分可自动测试和交互式代码
- 提供具体的测试方法和示例

**提示词设计**：
```markdown
## 测试策略

**可自动测试的代码**（无交互、无GUI）：
- 纯函数/类：直接运行 `python script.py`
- 有测试文件：运行 `pytest` 或 `python -m unittest`

**交互式代码**（包含 input(), GUI, 需要用户输入）：
- ❌ 不要直接运行（会卡住等待输入）
- ✅ 创建独立的测试文件，测试核心函数
- ✅ 或者说明"代码包含交互式输入，已创建完成，请手动运行测试"

**如何识别交互式代码**：
- 包含 `input()` 函数调用
- 包含 GUI 库（tkinter, PyQt, etc.）
- 包含无限循环 `while True:` 且没有明确退出条件
```

**面试话术**：
> "我在提示词中加入了测试策略指导。很多 Agent 系统会盲目运行所有代码，导致遇到 input() 时卡住。我教 Agent 识别交互式代码，并选择合适的测试方法。这是一种'教 Agent 如何思考'的提示词设计。"

**优势**：
- ✅ 避免 Agent 卡住
- ✅ 提高测试覆盖率
- ✅ 更好的用户体验

## 🔥 提示词工程的难点

### 1. **如何让 LLM 遵守约束**

**问题**：
- LLM 有时会忽略提示词中的约束
- 特别是在上下文很长时，约束可能被"遗忘"

**解决方案**：
```markdown
# 技巧 1: 反复强调
**禁止编造结果**：你绝对不能编造、假设或猜测任何执行结果。
...
**验证要求**：任何声称"成功"的回答，都必须有工具调用记录作为证据。

# 技巧 2: 使用示例
- ❌ 错误：说"我将创建一个文件"
- ✅ 正确：直接调用工具执行操作

# 技巧 3: 使用强制性语言
你必须【只输出纯 JSON】，不要有任何其他内容。

# 技巧 4: 使用 Reflection 检查
Reflection 会检查 Actor 是否遵守了约束
```

**面试话术**：
> "让 LLM 遵守约束是提示词工程的核心难点。我使用了多种技巧：反复强调、使用示例、强制性语言、Reflection 检查。关键是理解 LLM 的行为模式，知道什么样的提示词更有效。"

---

### 2. **如何平衡提示词长度和效果**

**问题**：
- 提示词太短：约束不够，LLM 行为不可控
- 提示词太长：成本高，可能超出 context window

**解决方案**：
```python
# 策略 1: 分层设计
# - 核心约束放在 SOUL.md（始终包含）
# - 详细指南放在工具描述中（按需包含）

# 策略 2: Prompt Caching
# - 稳定的提示词缓存，降低成本
# - 可以使用更长的提示词

# 策略 3: 动态注入
# - 根据当前状态注入相关的提示词
# - 不需要一次性包含所有提示词
```

**面试话术**：
> "提示词长度是一个权衡。我通过分层设计、Prompt Caching、动态注入来解决。核心约束始终包含，详细指南按需包含，稳定内容缓存降低成本。这样既保证了效果，又控制了成本。"

---

### 3. **如何处理不同模型的差异**

**问题**：
- 不同模型对提示词的理解不同
- 同一个提示词在不同模型上效果差异很大

**解决方案**：
```python
# 策略 1: 使用通用的提示词格式
# - 避免使用特定模型的特殊语法
# - 使用标准的 JSON 格式

# 策略 2: 提供多层防护
# - 提示词 + 解析器 + Guard + 验证 + 重试
# - 即使提示词失效，其他层也能兜底

# 策略 3: 配置驱动
# - 不同模型可以使用不同的提示词
# - 通过配置文件切换
```

**面试话术**：
> "不同模型对提示词的理解差异很大。我的策略是使用通用格式、多层防护、配置驱动。这样即使切换模型，系统也能正常工作。这是一种'防御性提示词设计'。"

## 🎯 提示词工程的最佳实践

### 1. **明确性 > 简洁性**

```markdown
# ❌ 不好：太简洁，容易误解
使用工具执行操作。

# ✅ 好：明确，不容易误解
当用户要求执行操作时，你必须：
1. 先调用工具执行操作
2. 等待工具返回真实结果
3. 基于工具返回的结果回答用户
4. 绝不能在没有调用工具的情况下声称"已完成"
```

### 2. **示例 > 描述**

```markdown
# ❌ 不好：只有描述
工具调用格式必须是 JSON。

# ✅ 好：有示例
**唯一正确格式**：
{"tool": "工具名", "arguments": {...}}

**错误格式示例**：
- ❌ 错误：<invoke name="run_cmd">...</invoke>
- ❌ 错误：```json\n{"tool": "run_cmd", ...}\n```
```

### 3. **约束 + 原因**

```markdown
# ❌ 不好：只有约束
禁止使用 echo 创建文件。

# ✅ 好：约束 + 原因
❌ **禁止使用 run_cmd + echo 创建文件**
- run_cmd 工具会拒绝执行 echo 文件操作
- 尝试使用会返回错误消息

✅ **必须使用 Write 工具**
- 直接创建文件并写入内容
- 如果文件已存在会报错，防止误覆盖
```

### 4. **分层组织**

```markdown
## 核心原则（最重要，始终遵守）
...

## 工具调用格式（次重要，调用工具时遵守）
...

## 工具使用指南（详细说明，按需参考）
...

## 回答模式（一般情况，灵活处理）
...
```
