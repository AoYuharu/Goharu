# 论文工程化分析

将一篇研究论文拆解为三层工程分析：先做原始提取，再做洞察与实验分析，最后做代码级模型分析及张量数据流。

## 默认立场

- 绝不编造论文内容、公式、引用、代码行为、超参数、指标或张量形状。
- 始终区分论文所述内容与你自己的推断。
- 若源材料不完整，返回带有可见缺口的局部分析，而非阻塞。
- 优先保证可追溯性：多用表格、显式标签和清晰映射，少用流畅散文。
- 若论文提供了 GitHub 链接，检查它；若没有，标记为无代码，并使用假数据重建继续推进。

## 接受的输入形式

论文 PDF 全文、文章文本、摘要+图表、粘贴摘录、阅读笔记、仓库 URL 或本地代码文件夹，中文或英文指令均可。

## 三阶段工作流

### 阶段一 —— 原始提取（`raw/`）

以论文标题为名创建工作区文件夹。在其中创建 `raw/` 文件夹，不做解读，仅做提取。

**1.1 论文各章节**
将论文的每个主要章节提取为 `raw/` 下的独立 Markdown 文件：

- `Abstract.md`
- `Introduction.md`
- `Related Work.md`
- `Methods.md`
- `Results.md`
- `Discussion.md`
- `Conclusion.md`
- `Appendix.md`（如有）

保留原文原样，不做润色或改写。若输入中缺少某个章节，仍创建该文件并标注 `[源材料中缺失]`。

**1.2 图表提取**
创建 `raw/images/`，对包含关键图表的每一页进行截图，按编号命名（如 `Figure_1.png`、`Table_2.png`）。如果论文是 PDF 格式，提取图表所在页；如果仅有文本，注明缺少哪些图表。

**1.3 引用文献**
创建 `raw/citations.md`。对论文中的每一条引用：
- 转换为适用于 LaTeX 的 APA 格式（`\usepackage{apalike}` 或 `biblatex-apa`）。
- 为每条文献标注出现位置（所在章节）及其作用（背景铺垫/基线来源/方法前身/数据集来源/指标定义/对比对象/局限性背景）。

格式如下：

```
## 引用清单

### [作者, 年份]
- **APA：** Author, A. (Year). Title. *Journal*, Vol, Pages.
- **出现章节：** Introduction, Related Work
- **引用作用：** 基线来源——本文与该工作在基准 X 上进行对比。
```

**1.4 GitHub 检查**
在论文中搜索 GitHub 或代码仓库 URL。结果记录在 `raw/repository_check.md`：
- 找到 URL → 记录下来，供阶段三使用。
- 未找到 → 标记 `CODE_ABSENT`，说明阶段三将采用假数据重建方案。

**1.5 总览**
创建 `raw/overview.md`，简要说明论文中每个提取部分涵盖的内容——每节一到两句话。只做描述，不做解读。

---

### 阶段二 —— 分析（`analyze/`）

在 `raw/` 同级创建 `analyze/` 文件夹。结合摘要、引言、相关工作三部分的证据提取洞察，然后审计实验。

**2.1 洞察提取 → `analyze/insight.md`**

用简洁直白的语言回答以下问题：

1. **论文要解决什么问题？**（来自摘要+引言）
2. **前人工作存在什么缺口？**（来自相关工作——已有方法缺少什么或未能做到什么）
3. **论文的核心创新是什么？**——相比所有引用的前人工作，唯一的新东西是什么
4. **这个创新是如何工作的？**——用不超过一段话，用通俗语言描述关键机制

结构如下：

```
## 要解决的问题

## 前人工作的缺口

## 核心创新

## 工作原理（通俗解释）
```

**2.2 实验分析 → `analyze/experiment_analysis.md`**

从结果和方法章节出发，评估以下内容：

1. **SOTA 判定** —— 论文是否声称达到最新最优？在哪些基准/数据集上？
2. **基线清单** —— 列出所有对比的基线方法。逐一判断：是标准基线、近期 SOTA 还是消融变体？
3. **改进点** —— 对每次基线对比，量化改进在哪里：
   - 性能（准确率、F1、BLEU 等）
   - 显存占用（VRAM、参数量、模型大小）
   - 计算量（FLOPs、训练时间、推理延迟）
   - 其他（数据效率、鲁棒性、泛化能力）
4. **消融实验** —— 消融了哪些组件？每个组件贡献了多少？
5. **统计严谨性** —— 是否报告了误差线、标准差或显著性检验？若没有，标记出来。
6. **可复现性风险** —— 超参数缺失、数据集划分不明确、专有数据集、未公开硬件。

结构如下：

```
## SOTA 声明

## 基线清单

## 各基线改进汇总

## 消融分析

## 统计严谨性

## 可复现性注意事项
```

**2.3 声明-证据映射 → `analyze/claim_evidence_map.md`**

对论文中的每一项核心声明，记录：

| 声明 | 类型 | 证据来源 | 支撑强度 | 备注 |
|---|---|---|---|---|
| ... | 性能/效率/鲁棒性/可扩展性 | 表格/图表/章节 | 强/部分/间接/不可验证 | ... |

---

### 阶段三 —— 模型与代码（`model/`）

在 `raw/` 和 `analyze/` 同级创建 `model/` 文件夹。

#### 情况一：存在 GitHub 仓库

将仓库克隆到 `model/upstream/`（保持原始代码不动）。

**3.1 入口分析 → `model/entrypoint/`**

创建 `model/entrypoint/entrypoint_analysis.md`，覆盖以下内容：

- **训练脚本位置** —— 主训练/评估入口文件的精确路径和行号
- **优化器** —— 使用的优化器、学习率、调度器、权重衰减
- **损失函数** —— 使用了哪个（些）损失函数，多损失时如何组合
- **TensorBoard / 日志** —— 记录了什么内容，记录频率
- **训练策略** —— `train/val/test` 划分，还是 `LOSO`（留一被试法）、`k-fold`；数据划分是如何实现的
- **Epoch 与超参数** —— batch size、epoch 数、早停条件、梯度裁剪、混合精度
- **评估协议** —— 计算的指标、评估频率、checkpoint 选择策略

将入口代码提取到 `model/entrypoint/` 中，作为独立的、带注释的 Python 文件，并附带 `README.md` 说明如何阅读。

**3.2 核心模块分析 → `model/core_modules/`**

从入口点出发，追踪完整的前向传播链条：

```
config/CLI → 模型构建 → 模块实例化 → forward → 输出头 → loss
```

对位于关键路径上、实现了论文方法中独特部分的每个模块：

- 将其提取为 `model/core_modules/<模块名>/` 下的可运行 Python 文件
- 附带 `README.md` 说明：
  - 该模块对应论文的哪个部分（章节、公式、图表）
  - 调用者/被调用者链条
  - 输入/输出张量形状

**3.3 数据流与张量形状 → `model/core_modules/data_flow.md`**

这是最核心的部分。利用提取的核心模块，追踪网络中每一步的张量形状。

方法：
1. 构造与论文输入规格匹配的合成输入数据（正确 dtype 和 shape 的假张量）。
2. 按顺序将假数据通过每一个核心模块。
3. 记录每一层的输入和输出张量形状。

输出格式 —— 形状追踪表：

```
## 数据流：[论文标题]

| 层/模块 | 输入形状 | 输出形状 | 论文出处 |
|---|---|---|---|
| 输入嵌入 | [B, L] | [B, L, D] | 第 3.1 节 |
| 编码器块 1 | [B, L, D] | [B, L, D] | 第 3.2 节, 公式 3 |
| ... | ... | ... | ... |
| 分类头 | [B, L, D] | [B, C] | 第 3.4 节 |
```

若能使用可视化库（如 `torchviz`、`netron` 或手绘图），给出可视化图；如果不能，仅凭形状追踪表即可。

每个形状条目必须标注证据状态：
- `已运行验证` —— 通过实际代码输入假数据运行并确认形状
- `从代码推断` —— 通过阅读代码推导形状，但未实际运行
- `从论文推断` —— 仅依据论文文字/公式推导形状

---

#### 情况二：不存在 GitHub 仓库

跳过克隆步骤，改为在 `model/` 中创建：

- `model/reconstruction.md` —— 尽力而为的重建方案：
  - 将关键公式翻译为 PyTorch/TensorFlow 伪代码
  - 根据论文描述推断各阶段的张量形状
  - 所有形状均显式标注为 `从论文推断`
- `model/attempted_implementation/` —— 使用假数据实现核心方法的最小可运行 Python 代码，附 `README.md` 说明实现了什么、假定了什么、哪些仍有疑问。

## 红线

- 绝不编造论文内容、公式、引用、代码行为、超参数、指标或张量形状。
- 除非通过实际运行验证，否则绝不声称代码可运行、可复现结果或与论文一致。
- 若没有论文中的明确支持，绝不夸大创新性或 SOTA 地位。
- 不隐藏失败状态——如果某个模块无法隔离、某个形状无法追踪、某个声明无法验证，明确说出来。
- 所有张量形状追踪均使用假数据/模拟数据，绝不要求真实数据集来验证模型结构。
- 每个推断都要清晰标注：`论文原文`（论文中明确记载）、`代码验证`（已实际运行）、`推断`（推理得出）、`未验证`（假设或猜测）。
