# 项目面试复习提纲

> 目标：快速讲清这个项目的核心链路。
> 项目本质：**一个 Server 调度、Client 执行、Web 展示的多设备手机自动化系统**。Server 负责推理和调度，Client 负责真机操作，Web 负责实时展示过程。

---

## 1. Server 与 Client 的状态同步

### 1.1 整体通信方式
这个项目不是单一协议，而是 **WebSocket + HTTP 混合通信**：

- **Client → Server（WebSocket）**：建立长连接、接收动作、回 ACK
- **Client → Server（HTTP）**：上报设备状态、上报 observe 结果
- **Server → Web（WebSocket）**：把 ReAct 过程实时推送到前端

可以直接这样说：

> **动作控制走 WebSocket，结果上报走 HTTP，前端展示也走 WebSocket。**

### 1.2 Server 和 Client 的关键同步链路

#### 设备在线 / 离线
- Client 连接 `/ws` 时，Server 会把它注册为设备连接。
- 如果该设备当前有任务，状态会被标记为 `BUSY`，否则标记为 `IDLE`。
- 断开连接时，Server 会清理该设备相关状态。

#### 设备状态同步
- Client 轮询本地设备列表。
- 每轮轮询结束后，通过 HTTP 把所有设备状态发给 Server。
- Server 维护设备在线、空闲、忙碌等状态，供调度和 Web 展示使用。

#### 动作执行同步
一个完整动作链路是：

1. Server 推理出下一步 action
2. Server 通过 WebSocket 发 `action_cmd` 给 Client
3. Client 先回 `ack`
4. Client 执行动作
5. Client 截图并通过 HTTP 上报 `observe_result`
6. Server 收到 observe 后继续下一轮推理

### 1.3 实时进度同步
Server 会把每一步的细粒度进度推送给 Web，典型顺序是：

```text
reason_complete
→ action_dispatched
→ waiting_ack
→ ack_received
→ waiting_observe
→ observe_received
```

这套顺序很重要，因为它对应前端 AgentWindow 的实时时间线。

### 1.4 一个很容易讲清楚的总结

> **Server 是“大脑 + 调度中心”，Client 是“设备执行器”。**
> Server 决定“做什么”，Client 负责“真的去点、去滑、去截图”，然后再把结果回传给 Server。

## 2. Server 中每个 device 的 session 和 ReAct 是如何处理的

### 2.1 每个 device 都有自己的会话和任务
Server 不是把所有设备混在一起处理，而是按 `device_id` 分开管理。

核心结构有两层：

- **session**：设备级持久会话，跨多次任务运行复用
- **run / task**：某一次具体执行

这里最关键的设计是：

- **`session_id`**：同一个设备长期复用，表示“这个设备的长期上下文”
- **`run_id`**：每次新任务新建，表示“这一次执行”

所以你可以这样理解：

> **session 是长期记忆，run 是一次执行实例。**

### 2.2 submit_task 时发生了什么
当 Web 侧发起一个新任务时，Server 会：

1. 检查该设备是否已有 session
2. 如果有，复用旧 `session_id`
3. 新建一个新的 `run_id`
4. 创建 `DeviceTask`
5. 把任务放入调度队列
6. 让该设备进入 ReAct 循环

这意味着：

- 同一个设备多次任务，**可以延续上下文**
- 但每次执行又有独立的 `run_id`，便于 ACK / observe / interrupt 精确归属

### 2.3 DeviceTask 里保存什么
`DeviceTask` 是单个设备当前任务的核心对象，里面保存：

- `device_id`
- `task_id`
- `instruction`
- `session_id`
- `run_id`
- `context`
- `react_records`
- `current_step`
- `phase`
- `status`

也就是说，**一个设备当前跑到第几步、上一轮看到了什么、现在处于 reason 还是 act 阶段**，都在这里。

### 2.4 ReAct 主循环怎么跑
每轮 ReAct 大致是：

1. 确保有 bootstrap 截图
2. 进入 `reason` 阶段
3. 调用模型，生成 reasoning + action
4. 进入 `act` 阶段
5. 通过 `ActionRouter` 下发动作
6. 等 ACK
7. 等 observe
8. 根据 observe 结果进入下一轮，直到 finish 或失败

一句话：

> **Server 每轮都在做“看屏幕 → 想一步 → 发动作 → 等结果 → 再想下一步”。**

### 2.5 ActionRouter 的作用
`ActionRouter` 可以理解成 **动作下发和回包匹配中心**。

它负责：

- 发送动作到 Client
- 生成 pending action
- 等待 ACK
- 等待 observe result
- 超时处理（`ack_timeout` / `observe_timeout`）
- 把实时阶段广播给 Web

所以调度器负责“策略和流程”，`ActionRouter` 负责“动作传输和结果匹配”。

### 2.6 Bootstrap observe 是什么
任务开始时，Server 会先拿一张初始截图，用来知道设备当前界面是什么。

这个初始截图对应 `step_number == 0`，它的特点是：

- 用于初始化观察
- **不进入正常 action pending 路由**
- 不会误触发某个动作回包逻辑

这个点在面试里很容易加分，因为说明你理解了“初始化观察”和“正式动作轮次”是分开的。

## 3. Agent 的上下文是如何管理的

### 3.1 上下文不是临时字符串，而是一个消息列表
Agent 上下文由 `DeviceTaskContext` 管理，本质上是：

- `system_prompt`
- `messages[]`

也就是典型的大模型消息结构。

### 3.2 上下文为什么按 device 维度保存
因为这个项目的核心对象是“设备会话”，不是“单条命令”。

所以：

- 一个设备跑完一个任务后
- 再接着跑下一个任务
- 仍然可以复用之前的上下文

这使得模型能延续“之前做到了哪里、当前设备大概处于什么状态”的记忆。

### 3.3 上下文从哪里恢复
如果内存里没有 context，Server 会从文件里的 `chat_history` 恢复：

- 读取历史消息
- 只把 `user / assistant / tool-use` 这类消息装回上下文
- 重建成当前设备的 `DeviceTaskContext`

所以它既有：

- **内存态上下文**：运行时快
- **文件态历史**：重启后可恢复

### 3.4 上下文何时更新
每轮推理时：

1. Server 取出 `system_prompt + 历史 messages`
2. 再拼上当前这轮的 user message（里面包含截图说明）
3. 调用模型
4. 把模型返回的 assistant 内容追加到 context
5. 同时写入 `chat_history`

所以它是“边跑边积累”的，不是每轮从零开始。

### 3.5 上下文如何防止无限变长
`DeviceTaskContext` 有 `truncate()` 逻辑：

- 按 token 数量控制长度
- 超过上限时，从最旧消息开始丢弃
- 至少保留最近消息

因此它是一个**可恢复、可截断的长会话上下文**。

### 3.6 前端恢复看到的是什么
Web 侧恢复 session 时，不只拿一段纯文本，而是拿到带丰富元数据的历史：

- `task_id`
- `session_id`
- `run_id`
- `step_number`
- `phase`
- `stage`
- `progress_message`

这样前端就能把历史重新拼成 timeline，而不是只显示聊天气泡。

### 3.7 一个面试里很好用的总结

> **这个项目的上下文管理是“设备级持久记忆 + 运行级精确归属”的组合。**
> 持久靠 `session_id` 和 `chat_history`，精确归属靠 `run_id`、`step_number`、`stage`。

## 4. Client 如何操控 Android，Server 给 Agent 的提示词是什么

## 4.1 Client 如何操控 Android
Android 侧核心是 `ADBAdapter`，也就是通过 ADB 命令去控制真机。

### 4.1.1 Client 收到动作后的流程
Client 收到 `action_cmd` 后会：

1. 校验消息版本
2. 先回 ACK
3. 选中对应设备的 adapter
4. 执行动作
5. 等界面稳定
6. 截图
7. 把截图和结果作为 `observe_result` 发回 Server

### 4.1.2 ADBAdapter 支持哪些动作
常见动作包括：

- `tap`
- `double_tap`
- `long_press`
- `swipe`
- `back`
- `home`
- `type`
- `launch`
- `wait`
- `finish`

它做的事情非常直接：

- 点击/滑动：转成坐标后调用 ADB
- 返回/Home：调用系统按键
- 输入文本：走输入法方案
- 截图：调用 `screencap`

### 4.1.3 截图怎么做
截图有多级回退：

1. `adb shell screencap -p` 到设备临时文件，再 `adb pull`
2. 失败就用 `adb exec-out screencap -p`
3. 再失败用第三套回退方式

这说明它不是只依赖单一路径，而是做了兼容。

### 4.1.4 文本输入怎么做
文本输入不是简单 `adb shell input text`，而是优先使用 **ADB Keyboard**：

- 先检查 ADB Keyboard 是否安装、启用、激活
- 必要时切换输入法
- 用 base64 广播 `ADB_INPUT_B64` 输入文本
- 输入后恢复原来的输入法

这个设计的好处是：

> **对 Unicode / 中文更友好，也比普通 input text 更稳定。**

## 5. Server 给 Agent 的提示词是什么

### 5.1 提示词的核心位置
Server 的核心 system prompt 在：

- `Server/src/services/react_scheduler.py:1635`

这个 prompt 明确告诉模型：

- 你是 Android 手机操作助手
- 每一步会收到截图和用户指令
- 要结合历史上下文继续任务
- 输出必须分成两部分：`<response>` 和 `<answer>`
- 动作必须用固定格式，比如 `do(action="Tap", ...)`

### 5.2 Prompt 规定了哪些动作
可用动作包括：

- `Launch`
- `Tap`
- `Swipe`
- `Type`
- `Back`
- `Home`
- `Wait`
- `finish`

并且明确要求：

- 坐标范围是 `0-999`
- 打开应用必须用 `Launch`
- 如果用户目标已经明显完成，要立即 `finish`
- 如果上一轮 observe 报错，不要机械重复同一错误动作

### 5.3 Screenshot Prompt 是怎么拼进去的
除了固定 system prompt，每轮还会根据截图情况再补一段动态提示：

- 一张图：分析当前界面
- 两张图：对比动作前后变化
- 如果前后截图完全一样：明确提醒模型“上一步动作没生效，不要重复”

这个设计很关键，因为它把“观察结果”显式转成了下一轮推理输入。

## 6. 你在面试里可以怎么总结

### 6.1 30 秒版本

> 这个项目是一个分布式手机自动化系统。Server 负责 ReAct 推理和任务调度，Client 通过 ADB/HDC/WDA 控制设备，Web 负责实时展示。
> 对 Android 来说，Server 根据截图和历史上下文调用模型生成动作，Client 收到动作后先 ACK，再执行、截图，并通过 observe_result 回传，Server 再进入下一轮推理，直到任务完成。

### 6.2 1 分钟版本

> 整个链路是：Web 下发任务给 Server，Server 为某个 device 创建或复用 session，然后进入 ReAct 循环。每一轮先看截图和历史上下文做 reason，再通过 ActionRouter 把 action 通过 WebSocket 发给 Client。Client 先 ACK，再执行 ADB 动作、截图，通过 HTTP 上报 observe_result。Server 收到后更新该 device 的 task、react_records 和 context，并把 `reason_complete → action_dispatched → waiting_ack → ack_received → waiting_observe → observe_received` 这些阶段实时推送到 Web。这样前端可以看到完整时间线，后端也能保持设备级长期上下文。

## 7. 最后记忆点

只要记住这 5 句话，基本就能应付大多数问题：

1. **Server 负责思考和调度，Client 负责真机执行。**
2. **动作走 WebSocket，observe 和 device status 走 HTTP。**
3. **每个 device 有长期 session，每次任务执行有独立 run。**
4. **上下文按 device 持久化，既在内存里，也能从 chat_history 恢复。**
5. **Android 控制核心是 ADBAdapter，输入靠 ADB Keyboard，截图有多级回退。**
