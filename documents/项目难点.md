## 🔥 一、技术难点（4 个核心）

### 难点 1：离线优先架构的数据一致性 ⭐⭐⭐⭐⭐

#### 问题背景

旅行场景网络不稳定，用户可能在无网络时：

- 添加账单
- 修改账单状态
- 退出旅行

**核心挑战**：

- 数据不丢失？
- 多设备数据最终一致？
- 重连后正确同步？

#### 解决方案

```kotlin
// 三层离线策略

// 1️⃣ 离线账单存储
RequestPackage.uploadBill(bill, object : ServerCallback<Int> {
    override fun onFailed(call: Call, e: IOException) {
        generatedBill.isOffline = true
        OfflineBillManager.addOfflineBill(generatedBill)  // 暂存本地
    }
  
    override fun onServerPermitted(data: Int?) {
        OfflineBillManager.removeOfflineBill(mTravelId, data)  // 同步后删除
    }
})

// 2️⃣ 离线状态队列
object OfflineBillStatusManager {
    val readyToSetStatusBox: MutableList<Int> = mutableListOf()
  
    fun addOfflineReadBills(billId: Int) {
        readyToSetStatusBox.add(billId)
    }
  
    fun update() {
        // 重连后同步未更新状态
        SocketIOManager.setBillsReadStatus(readyToSetStatusBox)
    }
}

// 3️⃣ 重连自动同步
socket?.on(Socket.EVENT_CONNECT) {
    OfflineHelper.setAttribute(false)
    OfflineBillStatusManager.update()  // 重连后同步
}
```

#### 面试话术

> 「离线同步是我花时间最多的模块。核心思路是**写操作本地化，读操作服务端优先**。
>
> 难点在于：
>
> - 要记录哪些数据是离线的（`isOffline` 标记）
> - 要维护一个待同步队列（`OfflineBillStatusManager`）
> - 重连后要按顺序同步，避免覆盖冲突
>
> 最终实现了用户无感知的离线体验。」

#### 考察点

- ✅ 异步编程能力
- ✅ 数据一致性思维
- ✅ 用户体验意识

### 难点 2：实时协作的状态同步 ⭐⭐⭐⭐⭐

#### 问题背景

多人同时操作同一旅行空间：

- A 添加账单，B 要即时看到
- C 退出旅行，其他人状态要更新
- 避免重复推送和状态不一致

#### 解决方案

```kotlin
// Socket.IO 事件驱动架构
SetSocketIOCallback<Bill>("billAdded", TAG) {
    BillManager.add(it)
    recyclerViewInitializer(true)
}

SetSocketIOCallback<TravelCard>("exitTravel", TAG) {
    TravelManager.update(it)
}

SetSocketIOCallback<TravelCard>("endTravel", TAG) {
    TravelManager.update(it)
    binding.addBills.visibility = View.GONE
}

SetSocketIOCallback<TravelCard>("someoneJoinedTravel", TAG) {
    TravelManager.update(it)
}

// 事件绑定管理
val bindingMap: HashMap<String, MutableList<String>> = HashMap()
// key: location (页面), value: [tag1, tag2, ...] (事件标签列表)
```

#### 面试话术

> 「实时协作的难点是**状态同步的时机和范围**。
>
> 我用 Socket.IO 的**房间概念**，每个旅行是一个房间，只有参与者能收到推送。
>
> 定义了 6 种事件类型：`billAdded`、`exitTravel`、`endTravel`、`someoneJoinedTravel` 等，每种事件触发对应的 UI 更新。
>
> 同时用 `bindingMap` 记录每个页面绑定的事件，避免内存泄漏。」

#### 考察点

- ✅ WebSocket/Socket.IO 理解
- ✅ 事件驱动架构
- ✅ 内存管理意识

### 难点 3：多人账单分摊算法 ⭐⭐⭐⭐

#### 问题背景

多人旅行账单场景复杂：

- A 帮 B、C 垫付，怎么算？
- 多人共同消费，怎么分摊？
- 转账和消费怎么区分？

#### 解决方案

```kotlin
// toolpackage 模块的 allocation 函数
val accountDatas: Map<String, AccountData> = allocation(currentTravelData.bills)

// 返回结构
data class AccountData(
    val OwnExpense: Double,      // 个人应付
    val CostForTeam: Double,     // 为他人垫付
    val Balance: Double          // 最终差额
)

// UI 展示
val ownExpense = String.format("%.2f", accountDatas[TravelNoteApplication.email]?.OwnExpense)
val costForTeam = String.format("%.2f", accountDatas[TravelNoteApplication.email]?.CostForTeam)

binding.mycostText.text = ownExpense
binding.payfortemText.text = costForTeam
```

#### 算法逻辑

```
遍历所有账单 → 
  如果是消费账单 → 
    垫付人 CostForTeam += 金额
    每个参与人 OwnExpense += 金额 / 人数
  如果是转账账单 →
    转账人 Balance -= 金额
    接收人 Balance += 金额
→ 输出每人净额
```

#### 面试话术

> 「账单分摊算法是核心业务逻辑。难点在于：
>
> - 要区分**消费账单**和**转账账单**
> - 要计算每个人**应付多少**、**垫付多少**
> - 最终得出**谁欠谁多少钱**
>
> 我用了一个 `Map<String, AccountData>` 结构，key 是邮箱，value 包含应付、垫付、差额三个字段。
>
> 遍历所有账单，累加计算，最后得出每个人的净额。」

#### 考察点

- ✅ 数据结构设计能力
- ✅ 业务逻辑抽象能力
- ✅ 数学思维

### 难点 4：多 Module 组件化架构 ⭐⭐⭐⭐

#### 问题背景

6 个模块（mainpage、travel、login、network、storage、toolpackage）：

- 模块间怎么通信？
- 怎么避免循环依赖？
- 怎么保证代码可维护性？

#### 解决方案

```kotlin
// 项目结构
TravelNote/
├── mainpage/      # 主页面模块
├── travel/        # 旅行详情页模块
├── login/         # 登录注册模块
├── network/       # 网络层模块
├── storage/       # 数据层模块
└── toolpackage/   # 工具类模块

// ARouter 路由
@Route(path = TravelNoteApplication.CashBookActivityRoute)
class MainTravelActivity : AppCompatActivity() {
    @Autowired
    @JvmField var travelId: Int = -1
}

// 跨模块跳转
ARouter.getInstance()
    .build(TravelNoteApplication.CashBookActivityRoute)
    .withInt("travelId", travelId)
    .navigation()
```

#### 依赖关系

```
mainpage → network, storage, toolpackage
travel   → network, storage, toolpackage
login    → network, storage
network  → storage
storage  → (无依赖)
toolpackage → storage, network
```

#### 面试话术

> 「多 Module 架构的难点是**解耦和通信**。
>
> 我用 ARouter 做路由层，模块间不直接依赖，通过路径跳转。
>
> 数据层（storage）和网络层（network）作为基础模块，其他模块依赖它们。
>
> 这样的好处是团队可以并行开发，我负责 mainpage 和 travel，队友负责 login，互不干扰。」

#### 考察点

- ✅ 架构设计能力
- ✅ 工程化思维
- ✅ 团队协作经验

## 💡 二、创新点（3 个核心）

### 创新点 1：旅行协作空间概念 ⭐⭐⭐⭐⭐

#### 传统方案痛点

现有旅行 APP 大多是**单人工具**：

- 账单、照片、行程分散在不同应用
- 结伴出行时信息不同步
- 需要手动同步给队友

#### 你的创新

```
┌─────────────────────────────────────────┐
│         旅行空间 (Travel Space)          │
├─────────────────────────────────────────┤
│  👥 参与者：A、B、C                      │
│  📊 账单：自动同步，多人可编辑           │
│  📍 轨迹：高德地图集成，实时位置共享     │
│  📷 照片：集中存储，智能聚类             │
│  📅 行程：共同规划，实时可见             │
└─────────────────────────────────────────┘
```

#### 面试话术

> 「TravelNote 的核心创新是**旅行空间**概念。
>
> 每次旅行创建一个独立空间，所有参与者共享：
>
> - 账单自动同步，支持多人编辑
> - 照片集中存储，智能聚类
> - 行程共同规划，实时可见
>
> 这解决了结伴出行时信息割裂的问题。」

#### 核心价值

- ✅ 多人实时协作
- ✅ 账单、轨迹、照片、行程四合一
- ✅ 覆盖行前规划→行中记录→行后回顾完整周期

### 创新点 2：智能账单系统 ⭐⭐⭐⭐⭐

#### 传统方案痛点

记账 APP 操作繁琐：

- 需要手动输入金额、分类、参与人
- 不支持多人分摊
- 无网络时无法使用

#### 你的创新

| 功能     | 传统方案 | TravelNote          |
| -------- | -------- | ------------------- |
| 分摊计算 | 手动计算 | 自动 AA 分摊        |
| 转账支持 | 不支持   | 长按添加转账        |
| 离线记账 | 不可用   | 本地存储 + 自动同步 |
| 实时通知 | 无       | Socket.IO 推送      |

#### 面试话术

> 「我设计了**智能账单系统**：
>
> 1. **自动分摊**：输入金额和参与人，自动计算每人应付
> 2. **转账区分**：长按添加转账，和普通消费分开统计
> 3. **离线记账**：无网络时也能添加，恢复后自动同步
> 4. **实时推送**：队友记账后，其他人即时收到通知
>
> 这比传统记账 APP 更适合多人旅行场景。」

#### 技术支撑

- ✅ Socket.IO 实时推送
- ✅ 离线队列 + 重连同步
- ✅ allocation 算法自动分摊
